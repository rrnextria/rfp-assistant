## Detailed Objective

Add the "Generate Bid Assessment Report" deliverable — a branded PDF or DOCX produced from a completed assessment, intended for bid-committee review. The renderer uses Jinja2 templates with tenant brand variables (logo, name, colours, header/footer text) and pulls the assessment's rolled-up scorecard plus all child rows (compliance, eligibility, risks, capability matches, exec summary).

Files persist on a local Docker volume (`/var/exports`) and are returned to the client as signed URLs with a TTL. Phase-2 swap to S3 or Azure Blob is a one-adapter change — kept abstract behind a `StorageBackend` interface. The phase ends with an Akkodis-branded sample PDF and DOCX usable as a real demo asset.

Success: From the scorecard, clicking "Export PDF" produces a downloadable, Akkodis-branded PDF containing the full assessment; the same for DOCX; both formats render correctly on a default Linux PDF reader and Microsoft Word respectively.

## Deliverables Snapshot

1. Jinja2 templates in `services/rfp-service/exports/templates/`: `report.html.j2` (for PDF), `report.docx.j2` (for DOCX) — both parameterised on tenant brand vars and assessment data.
2. `services/rfp-service/exports/render.py` — produces PDF (via `weasyprint`) and DOCX (via `python-docx-template`) from the Jinja2 templates.
3. `services/rfp-service/exports/storage.py` — `StorageBackend` interface + `LocalFilesystemBackend` v1 implementation persisting to `/var/exports`.
4. Endpoints: `POST /rfps/{id}/assessments/{aid}/export` and `GET /rfps/{id}/assessments/{aid}/exports`.
5. `docker-compose.yml` gains a named volume `rfp_exports` mounted at `/var/exports` in `rfp-service`.
6. Frontend `AssessmentExportDialog.tsx` — format chooser + preview of brand vars + generate + download.
7. Akkodis-branded sample PDF and DOCX checked in as demo assets (or generated by a fixture test).

## Scope

- In Scope:
  1. Two Jinja2 templates (HTML for PDF, DOCX).
  2. Renderer modules for both formats.
  3. `StorageBackend` interface + local filesystem implementation.
  4. Signed-URL TTL mechanism (HMAC-signed token, configurable TTL, default 1h).
  5. Export endpoints + listing.
  6. Docker volume for export persistence.
  7. Frontend export dialog.
- Out of Scope:
  1. S3 / Azure Blob backends — interface exists, implementation deferred to Phase-2 backlog.
  2. Watermarking — schema reserves a field, no rendering in v1.
  3. Custom per-tenant template variants — single canonical template per format.
  4. Multi-page TOC / cross-references inside the document.
  5. Live preview in the dialog — only the brand vars preview.

## Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| `weasyprint` system-dep heaviness (cairo, pango) bloats the rfp-service image | Slower builds, bigger image | Pin `weasyprint` to a known-good version; cache system deps in the Dockerfile; alternatively use `WeasyPrint` headless service if image bloat is unacceptable. |
| Brand logo file missing for a tenant | Render fails | Renderer falls back to text-only header when `brand.logo_url` is unset or unreadable. |
| Large assessments produce huge PDFs (50+ pages) | Slow render, big files | No per-page cap; document in the dialog UI; recommend manual print-to-PDF if needed. |
| Signed URL exposes export to unauthenticated downloaders | Data leakage | Token is HMAC-signed with `JWT_SECRET`-derived key; verified on GET; returns 401 if expired or tampered. |
| Local filesystem doesn't survive container restart without volume | Lost exports | Named Docker volume `rfp_exports` declared in `docker-compose.yml`; `assessment_exports` row stays referenceable even if the file is gone (404 on download). |
